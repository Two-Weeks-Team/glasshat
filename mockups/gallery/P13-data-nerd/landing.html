§1Methodology

We treat the Gemini 3 Hackathon (Dec 2025 — Feb 2026, gemini3.devpost.com) as our held-out evaluation corpus. The crawl captured the first 503 gallery submissions across 21 pages, of which 13 carry the Devpost Winner badge (1 Grand Prize, 1 Second, 1 Third, 10 Honorable Mentions). The task: predict a project’s probability of being a winner from public artefacts only (Devpost description + GitHub metadata + built_with tags).

1.2 ECE definition

We use the standard Naeini et al. (2015) definition with M=15 equal-width bins: ECE = Σᵢ (|Bᵢ|/n) · |acc(Bᵢ) − conf(Bᵢ)|. Confidence intervals via non-parametric bootstrap (1,000 resamples of the 503 rows with replacement, seed 42). All five reported metrics use the same resamples for paired inference.

1.3 Honest baselines

Vanilla judge sees the same prompt and rubric Glasshat synthesizes — the only differences are (a) the six-hat decomposition, (b) the Qdrant anchor retrieval, (c) the audit loop. No prompt-engineering advantage is given to Glasshat. Both systems are pinned to Gemini 3.1 Pro at 2026-04-22 snapshot for the core judge call.

3.4 Anchor-cosine fidelity

For each submission, we measure cosine similarity between the submission’s pitch+repo embedding and its 12 retrieved anchors from past_evals. Median cosine = 0.78 [IQR 0.71, 0.84], n=503×12 = 6,036 pairs. Distribution is left-skewed with a long tail of weak matches that correctly trigger the audit loop (cosine < 0.60 fires weak-anchor evaluator in 41 cases).

§4Limitations

We are not selling a finished product; we are reporting a measured improvement on one public corpus. Specifically:

limitationimpactmitigation
Corpus is gallery-truncated (503 of ~700 entries) Possible selection bias on early-page projects Page-stratified bootstrap shows ρ stable ±0.04
Winner label is binary, not ordinal Spearman uses Devpost prize order only for n=13 Reported with wide CI; supplementary ranking via judge-only
No inter-annotator agreement available Cannot separate “judge bias” from “ground-truth noise” Phoenix annotation API records human override deltas in production
Single LLM family (Gemini 3 series) Calibration gain may not transfer to GPT/Claude judges Future work: cross-family replication on same 503 corpus
Hackathon-judge task ≠ general LLM-judge task Generalisation to other rubrics unproven RubricSynthesizer abstraction designed for this; not yet measured
What this report does not claim: that Glasshat is unbiased, that calibration gains persist on out-of-distribution rubrics, or that 38.4% ECE reduction implies 38.4% better decisions. All claims are scoped to predicting Devpost outcomes on the 503 Gemini 3 sample under the BMAD-derived rubric.
