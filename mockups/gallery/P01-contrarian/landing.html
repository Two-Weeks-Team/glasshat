Audit of an A.I. Jury
“Glasshat doesn't just judge projects. It audits the judge.”

I   What is filed here

On the docket is a working, end-to-end evaluation pipeline that ingests a pitch deck (PDF), a code repository (GitHub), and the evaluator's own rules. A RubricSynthesizer agent parses those rules into a per-evaluation rubric. A six-perspective AI panel1 scores each submission, with every sub-score grounded in vector-retrieved evidence and anchored to comparable past evaluations. non-chatbot. drop-zone first.

When a hat over-confidences itself, three independent auditors — two of them sub-agents living inside Arize Phoenix, the third the Black hat itself — file a counter-claim. The Blue planner opens an MCP channel to Phoenix at runtime, retrieves the drift statistic, asks Qdrant's Recommendation API for anti-pattern anchors, and the score self-corrects on screen. The 3-D evaluation graph reshapes.2

The defense then displays the same submission scored under two synthesized rubrics, side-by-side. Caption, on the record: “Correct rubric-aware variance, not bias.” Fairness, it turns out, is rubric-relative — and we can prove it.

IV   How (the loop, briefly)

Glasshat ingests inputs that aren't prompts — a PDF deck, a repo URL, an evaluator's official rules. The deck and code are chunked and embedded into Qdrant under both dense and sparse vectors; retrieval is hybrid with RRF fusion. A Blue planner (Google ADK on Cloud Run, Gemini 3.1 Pro with thinking_level=high) emits an inspectable plan; the user approves it (human gate 1).

Six perspective sub-agents run in parallel under ADK ParallelAgent. Each emits a structured finding with claims, evidence references, and a sub-score. The Black hat must cite precedent — enforced via plugin callback.

An ADK LoopAgent with max_iterations=2 performs the audit. Three independent detectors (Phoenix Online Eval LLM-judge, Phoenix Python custom evaluator, Black-hat counter-claim) feed a single InconsistencyDetector. On a flag, the PhoenixConsultantAgent makes four parallel MCP calls — three to Phoenix, one to Qdrant — returning drift statistics and anti-pattern anchors.

The ScoreCalibrationAgent applies a conservative pull: new_score = clip(predicted − 0.8 · mean_delta, anchor_p25, anchor_p75), capped at ±2.0. On convergence it sets escalate=True and the loop exits. The user can override any final score with a written reason (human gate 2), recorded as a Phoenix Annotation that feeds future calibration. The loop closes.

V   Why this is worth your time

Every other 2026 hackathon submission is going to look the same: dark background, glass card, neon gradient, Three.js orb. The cohort agreed, apparently in private, to ship identical landings. We refused. Glasshat is an audit product, so we built it like an audit report — black ink on cream paper, page numbers, footnotes, marginal corrections, rubber stamps. The form is the function.

There is no chat box. There is no “hello, how can I help.” There is a drop zone, a plan, six hats, an audit loop, and a signed scored report. The artifact you receive at the end is hash-anchored: input hashes, repo SHA, rubric version, model versions, web-search snapshot, per-criterion score with evidence and precedent IDs, every human override. If you don't trust the score, you can replay it.
